## 项目定位

Legal Skills 是一套开源的中文法律工作 Skill 集合，目标不是提供一个“万能法律提示词”，而是把真实法律服务中容易被模型忽略的流程、校验、分工和交付边界，拆解为可由 AI Agent 执行和复核的工作单元。

在这个项目里，Agent 不是被鼓励直接给出结论，而是被要求按法律工作习惯推进：

- 先识别任务类型，再选择对应的专业 Skill；
- 先读取材料、复查关键信息，再进入分析或起草；
- 涉及法规、案例、政策或规范性文件时，必须暴露来源和校验状态；
- 材料不足、OCR 存疑、事实冲突、期限风险和授权不明时，必须向使用者提示；
- 生成正式 Word 文书前，必须经过出稿前审查和导出质量门；
- 对外输出始终保留专业人士复核空间，避免把模型推断包装成已核验事实或法律意见。

因此，Legal Skills 更接近一个“法律 Agent 工作台的规则层”：它把律师、法务、法律科技团队日常处理事项时的流程经验，沉淀为 Markdown Skill、流程协议、文书模板、检查清单和少量本地脚本。

> **重要提示：本项目不提供法律意见。**
> 本项目输出的任何内容都应视为供律师、法务或合格专业人士复核的工作草稿。真实法律事项必须由专业人士基于完整事实、有效授权、现行法律、可核验案例和具体语境作出独立判断。

## 适合谁使用

Legal Skills 适合以下使用者：

| 使用者 | 可以获得什么 |
|---|---|
| 律师 | 将咨询回复、案件分析、文书起草、材料复查、庭前准备、合同审查等工作拆成稳定流程，减少遗漏 |
| 企业法务 | 搭建产品合规、劳动合规、合同审查、监管动态监测和内部法律支持的 Agent 工作流 |
| 法律科技开发者 | 参考 Skill 目录结构、路由规则、质量门和 DOCX 导出链路，构建自己的法律 Agent 应用 |

| 法学院、研究者和教学团队 | 观察法律知识、流程控制、来源校验和文书生成如何转化为 Agent 可执行规范 |

## 解决的问题

法律工作并不只是“给模型一个问题，让模型回答”。真实业务通常同时包含事实、证据、程序、期限、授权、检索、格式和交付对象的多重约束。

Legal Skills 试图解决这些具体问题：

- **任务路由不稳定**：同一句“帮我看看这个案子”，可能需要咨询回复、诉讼分析、证据审查、法规检索、文书起草或案件管理。项目通过 `法律工作总控` 统一识别任务并分发到专业 Skill。
- **材料读取不透明**：法律结论依赖文件内容。项目强调文件读取、OCR 识别、关键信息抽取、完整性复查和存疑项披露。
- **法规案例容易失真**：模型可能凭记忆引用失效、错号或不存在的法条和案例。项目要求对法规、案例、政策、Wiki 内容和网页来源进行校验或明确标注未校验状态。
- **法律文书容易跳步骤**：起诉状、答辩状、代理词、合同、报告、授权文件等正式交付物不能只看“写得像”。项目设置出稿前审查、身份占位、事项路径、结构完整性和 DOCX 导出检查。
- **本地业务系统难以迁移**：律师或团队通常有自己的客户台账、文件夹结构、云文档、OCR、案例库和知识库。项目将私有路径和身份信息抽离，保留可替换的配置边界。
- **专业边界容易模糊**：项目反复强调输出只是工作草稿，不替代专业判断、客户授权、律师责任和最终复核。

## 核心设计

Legal Skills 由三层组成：

```text
用户任务
  │
  ▼
法律工作总控
  - 语义路由
  - 事项隔离
  - 材料读取复查
  - 法规/案例校验
  - 来源边界
  - 输出底线
  │
  ▼
专业业务 Skill
  - 民事诉讼
  - 刑事辩护
  - 劳动争议
  - 破产程序
  - 合同与交易
  - 产品法务与合规
  - 法律咨询与检索
  │
  ▼
交付质量门
  - 出稿前审查
  - 文书模板与导出
  - DOCX 结构检查
  - 缺口与风险提示
```

### 1. 总控层

`法律工作总控` 是统一入口，负责在法律任务开始时建立共同规则。它不替代子 Skill 的专业判断，而是约束所有子 Skill 的最低质量线，包括：

- 根据任务语义匹配具体 Skill；
- 对具体事项进行事项隔离；
- 对文件、图片、网页、法规和案例触发读取或校验规则；
- 要求输出来源边界和未完成事项；
- 对正式 Word 交付触发 `法律文书出稿前审查` 和 `法律文书模板与导出`。

### 2. 业务层

业务 Skill 对应具体法律工作场景，例如合同审查、民事一审诉讼、刑事辩护、劳动争议、破产程序、产品法务、广告合规、监管监测和法律咨询。每个 Skill 通常包含：

- `SKILL.md`：触发条件、适用范围、执行步骤、确认点、禁止事项；
- `references/`：流程协议、方法论、检查表、专项规则；
- `templates/`：文书模板、报告框架、清单或交付物结构；
- `scripts/`：可选的本地自动化脚本。

### 3. 交付层

正式文书、报告、清单、合同和授权文件不直接从业务 Skill “一键结束”。项目将交付拆成两道质量门：

- `法律文书出稿前审查`：检查事实来源、结构完整性、身份占位、法规校验、事项路径和交付风险；
- `法律文书模板与导出`：将审查后的语义 HTML 按 profile 导出为本地 DOCX，并做结构体检。

目前委托授权文件包仅保留 `授权委托书` 和 `法定代表人身份证明书` 两类公开模板。

## 能力地图

| 模块 | 代表 Skill | 主要能力 |

| 总控与共享协议 | `法律工作总控` | 任务路由、事项隔离、文件读取复查、法规案例校验、输出边界、质量门调度 |

| 刑事辩护 | `刑事辩护总调度`、`侦查阶段辩护`、`审查起诉阶段辩护`、`一审阶段辩护` | 按诉讼阶段路由，覆盖会见、阅卷、取证、庭前会议、量刑和特殊案件分支 |

| 合同与交易 | `合同起草`、`合同审查`、`委托合同管理` | 合同生成、合同审查、风险清单、版本修订、授权文件和归档提示 |

| 文书交付 | `法律文书出稿前审查`、`法律文书模板与导出`、`审理报告`、`民事判决书` | 出稿审查、DOCX 导出、法院文书样式、报告结构和交付前体检 |

## 典型工作流

### 案件材料分析

```text
用户上传案件材料
  │
  ▼
总控识别为民事/刑事/劳动/破产等事项
  │
  ▼
逐一读取 PDF、Word、图片、表格或网页
  │
  ▼
输出读取复查摘要
  │
  ▼
提取主体、金额、日期、案号、证据和争议焦点
  │
  ▼
进入初步法律分析、证据缺口或文书起草
```

### 合同审查

```text
合同文本或文件
  │
  ▼
合同审查 Skill
  │
  ▼
交易背景、主体、金额、履行、违约、解除、争议解决条款检查
  │
  ▼
输出风险等级、修改建议、替换条款和需确认问题
```

### 正式文书交付

```text
业务 Skill 生成正文或语义 HTML
  │
  ▼
法律文书出稿前审查
  │
  ├─ PASS / FIXED_PASS：进入导出
  └─ NEEDS_*：退回业务修订、用户确认或补充材料
  │
  ▼
法律文书模板与导出
  │
  ▼
DOCX 文件与结构体检结果
```

## 快速开始

### 方式一：作为普通 skills 目录安装

```bash
git clone https://github.com/pa1nrui1/legal-skills.git
mkdir -p ~/.codex/skills
rsync -a legal-skills/skills/legal/ ~/.codex/skills/legal/
```

安装后，在 Codex 或其他支持本地 skills 的 Agent 中直接调用：

```text
法律工作总控 帮我判断这个任务应该走哪个法律工作流。
```

```text
合同审查 审核这份合同，按风险等级输出修改建议。
```

```text
法律咨询助手 帮我回复这个客户咨询，保持微信式简短自然。
```

```text
民事一审诉讼 读取这个案件材料，先做读取复查摘要，不要直接生成法律意见。
```

### 方式二：作为 Codex plugin 安装

```bash
codex plugin marketplace add pa1nrui1/legal-skills --sparse .codex-plugin --sparse skills
```

## 本地配置

本仓库已删除作者个人执业身份、联系方式和作者本机路径。用于真实业务前，建议按自己的组织环境配置：

- 律师姓名、律所名称、地址、电话、邮箱等身份占位；
- 工作区、客户台账、系统记录、业务文件区等本地保存规则；
- 法规检索、案例库、OCR、飞书或其他外部服务凭证；
- 是否允许 Agent 写入系统记录、日历、云文档或其他外部系统；
- 正式文书的字号、页边距、抬头、签章、编号和归档规范；
- 团队内部对客户沟通、风险提示、复核和审批的权限边界。

建议把真实业务配置保存在本机私有文件、私有仓库或环境变量中，不要提交到公开仓库。

## 安全边界

Legal Skills 默认强调以下边界：

- 不把模型记忆包装成文件事实、现行法规或已核验案例。
- 不在未读取材料、未完成复查、未校验法规时生成正式法律意见。
- 不静默写入台账、系统记录、飞书文档或日历。
- 不把客户材料、案卷、OCR 中间结果、API key 或商业数据库凭证提交到仓库。
- 不替代律师、法务或专业人士对事实、证据、法律适用和最终文本的判断。
- 不承诺诉讼、仲裁、行政处理、交易谈判或监管沟通结果。

## 开源范围

本仓库发布的是 Skill 规则、流程、模板和脚本，不包含：

- 作者个人身份信息；
- 作者本机路径；
- 真实客户材料；
- 业务台账或系统记录；
- 飞书文档；
- 私有数据库凭证；
- 迁移备份目录。

开源整理时已将文书模板中的身份信息改为通用占位符，并将外部服务凭证改为显式环境变量配置。

## 适配其他 Agent

这些 Skill 本质上是 Markdown 工作流说明和少量脚本。即使不使用 Codex，也可以把 `skills/legal` 目录复制到其他支持本地规则、工作流或知识文件的 Agent 工具中，再根据该工具的加载机制做适配。

适配时建议保留：

- `SKILL.md` 的触发条件和执行顺序；
- `法律工作总控/references/` 中的共享协议；
- `templates/` 中的文书结构；
- 出稿前审查和文书导出的质量门；
- 对法规、案例、事实材料和 OCR 结果的来源披露要求。

如果接入自己的知识库、检索系统、DMS、CRM、云文档或办公自动化系统，建议先将外部写入动作设计为显式确认，再逐步放开自动化权限。

## 适合二次开发的方向

- 为不同法域或不同执业领域建立独立 Skill 包；
- 接入团队自己的法规库、案例库、合同库和知识库；
- 将出稿前审查结果转化为结构化 JSON，接入审批流；
- 增加更多文书 profile，例如仲裁文书、行政复议文书、内部合规报告；
- 将 OCR、法规检索、案例检索和 DOCX 导出替换为团队自有服务；
- 为高频咨询、合同条款和诉讼节点建立评估集，持续测试 Skill 表现。
