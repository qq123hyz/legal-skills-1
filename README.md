## 关于项目

**Legal Skills** 是一套开源的中文法律工作 Skill 集合。它不是“万能法律提示词”，而是把真实法律服务中容易被模型忽略的流程、校验、分工和交付边界，拆解为可由 AI Agent 执行和复核的工作单元。

Agent 在本项目中不会被鼓励直接给出结论，而是被要求按法律工作习惯推进：

- 先识别任务类型，再选择对应专业 Skill；
- 先读取材料、复查关键信息，再进入分析或起草；
- 涉及法规、案例、政策或规范性文件时，必须披露来源和校验状态；
- 材料不足、OCR 存疑、事实冲突、期限风险和授权不明时，必须提示；
- 生成正式 Word 文书前，必须经过出稿前审查和导出质量门；
- 对外输出始终保留专业人士复核空间，避免把模型推断包装成已核验事实或法律意见。

> **重要提示：本项目不提供法律意见。**
> 本项目输出的任何内容都应视为供律师、法务或合格专业人士复核的工作草稿。真实法律事项必须由专业人士基于完整事实、有效授权、现行法律、可核验案例和具体语境作出独立判断。

## 项目概述

法律工作并不只是“给模型一个问题，让模型回答”。真实业务通常同时包含事实、证据、程序、期限、授权、检索、格式和交付对象的多重约束。

Legal Skills 试图解决这些具体问题：

1. **任务路由不稳定**：同一句“帮我看看这个案子”，可能需要咨询回复、诉讼分析、证据审查、法规检索、文书起草或案件管理。
2. **材料读取不透明**：法律结论依赖文件内容，必须有文件读取、OCR 识别、关键信息抽取、完整性复查和存疑项披露。
3. **法规案例容易失真**：模型可能凭记忆引用失效、错号或不存在的法条和案例，必须对法规、案例、政策、Wiki 内容和网页来源进行校验或标注未校验状态。
4. **法律文书容易跳步骤**：起诉状、答辩状、代理词、合同、报告、授权文件等正式交付物不能只看“写得像”，必须有出稿前审查、事项路径、结构完整性和 DOCX 导出检查。
5. **本地业务系统难以迁移**：律师和团队通常有自己的客户台账、文件夹结构、云文档、OCR、案例库和知识库，本项目将私有路径和身份信息抽离，保留可替换配置边界。

### 核心特点

- **法律工作总控**：统一路由、事项隔离、材料读取复查、法规案例校验和来源边界。
- **专业业务 Skill**：覆盖民事诉讼、刑事辩护、劳动争议、破产程序、合同、产品法务、监管合规、法律咨询和法律研究。
- **正式交付质量门**：正式 Word、飞书正式交付物、业务流程图、修订稿、图片交付包均有明确终检规则。
- **真实 DOCX 红线能力**：合同审查支持生成带真实修订痕迹和批注的 Word 审核修订稿。
- **模板克隆导出能力**：要素式起诉状可走 DOCX 母版克隆填充，保留复杂表格、合并单元格、空白页和版式结构。
- **本地化友好**：作者个人信息、本机路径、客户台账和外部凭证均以占位或配置边界处理。

## 典型工作流

#### 案件材料分析

```text
用户上传案件材料
  │
  ▼
法律工作总控识别事项类型
  │
  ▼
逐一读取 PDF、Word、图片、表格或网页
  │
  ▼
输出读取复查摘要
  │
  ▼
提取主体、金额、日期、案号、证据和争议焦点
  │
  ▼
进入初步法律分析、证据缺口或文书起草
```

#### 合同审查

```text
合同文本或文件
  │
  ▼
确认审查立场：甲方 / 乙方 / 中立
  │
  ▼
合同审查 Skill
  │
  ▼
输出风险等级、修改建议、替换条款和需确认问题
  │
  ▼
如用户要求 Word 修订模式：生成 redline-plan.json
  │
  ▼
执行真实 DOCX 修订/批注并完成红线 QA
```

#### 正式文书交付

```text
业务 Skill 生成正文、语义 HTML 或要素式填充数据
  │
  ▼
法律文书出稿前审查
  │
  ├─ PASS / FIXED_PASS：进入导出
  └─ NEEDS_*：退回业务修订、用户确认或补充材料
  │
  ▼
法律文书模板与导出
  ├─ 普通线性文书：draft_checked.html -> html_to_docx.py
  └─ 要素式起诉状：complaint-data.json + fill-plan.json -> fill_docx_template.py
  │
  ▼
DOCX 文件、质控报告与结构体检结果
```

## 快速开始

```bash
git clone https://github.com/pa1nrui1/legal-skills.git
mkdir -p ~/.codex/skills

```

安装后，在 Codex 或其他支持本地 skills 的 Agent 中直接调用：

```text
法律工作总控 帮我判断这个任务应该走哪个法律工作流。
```

```text

```


## 本地配置

本仓库不包含作者本机业务路径、真实客户材料和外部服务凭证；涉及执业身份、联系方式和机构信息的内容可按自己的组织环境替换或通过环境变量配置。用于真实业务前，建议配置：

- 律师姓名、律所名称、地址、电话、邮箱等身份占位；
- 工作区、客户台账、系统记录、业务文件区等本地保存规则；
- 合同起草所需的本地合同模板库路径；
- 法规检索、案例库、OCR、飞书或其他外部服务凭证；
- 是否允许 Agent 写入系统记录、日历、云文档或其他外部系统；
- 正式文书的字号、页边距、抬头、签章、编号和归档规范；
- 要素式起诉状 DOCX 母版目录：`LEGAL_TEMPLATE_ROOT`；
- 自定义模板克隆 manifest：`LEGAL_TEMPLATE_CLONE_MANIFEST`；
- 可选 DOCX 渲染脚本：`LEGAL_DOCX_RENDER_SCRIPT`；
- 团队内部对客户沟通、风险提示、复核和审批的权限边界。

建议把真实业务配置保存在本机私有文件、私有仓库或环境变量中，不要提交到公开仓库。

## 安全边界

Legal Skills 默认强调以下边界：

- 不把模型记忆包装成文件事实、现行法规或已核验案例。
- 不在未读取材料、未完成复查、未校验法规时生成正式法律意见。
- 不静默写入台账、系统记录、飞书文档或日历。
- 不把客户材料、案卷、OCR 中间结果、API key 或商业数据库凭证提交到仓库。
- 不替代律师、法务或专业人士对事实、证据、法律适用和最终文本的判断。
- 不承诺诉讼、仲裁、行政处理、交易谈判或监管沟通结果。

## 适配其他 Agent

这些 Skill 本质上是 Markdown 工作流说明和少量脚本。即使不使用 Codex，也可以把 `skills/legal` 目录复制到其他支持本地规则、工作流或知识文件的 Agent 工具中，再根据该工具的加载机制做适配。

适配时建议保留：

- `SKILL.md` 的触发条件和执行顺序；
- `法律工作总控/references/` 中的共享协议；
- `templates/` 中的文书结构；
- 出稿前审查和文书导出的质量门；
- 对法规、案例、事实材料和 OCR 结果的来源披露要求。
